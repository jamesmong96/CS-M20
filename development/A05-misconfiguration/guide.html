A05: Security Misconfiguration

Please read the following scenario

You found the login page of the student intranet a bit suspicious and it is very likely you can find a way to login without your own credential.

Click here to enter the Login Page

Before you start exploiting the console page, please read what is security misconfiguration. There are three misconfigurations exist in the page, find all of them to gain access to the console. You can explore the directory by yourself and search for any traces of the misconfigurations.

Vulnerability

There are three misconfigurations lead to this vulnerability, you can gain the account information by taking advantages of these misconfigurations. During development process, it is completely normal to have a lot of available resources in the directories. Some of them can aid the development and facilitate the software development cycle. However it is also very common to see these kinds of resources left accessible to public in production environment. As a result, attackers can search for these exposed sensitive information and gain access to the application.

Exploitation
  1. Find the symbolic link of this directory by going to /A05-misconfiguration/.
  2. You should notice there is a testing-account.txt, which very likely to have some account information.
  3. But you should find that the username and password in it cannot login successfully.
  4. Press F12 and click on the Console tab.
  5. You should notice that there are some debug messages when you submit the username and password.
  6. Enter the username and password you found again, there is a debug message like this: This testing account is deprecated, please find the updated account in ../resources/A05/testing-account.txt.
  7. Follow the link here: ../resources/A05/testing-account.txt.
  8. Another set of username and password is found in this file.
  9. You can login to the page successfully with the newly found username and password!
Explanation

This vulnerability occurs because there are three misconfigurations in the web server and the page. These misconfigurations let the attackers discover all the content in this directory and find the correct account information in the server. Listing all the content on the directory can be done intensionally when the website owners want to increase the page rank of the websites. But before doing so, it is very important that there are no unnecessary files on the directory and other directories are not affected by it. Developers should also avoid using error messages which are too informative and contain information that the users don't need to know in the production environment. This will only provide more details of the system and expose sensitive information, such as variable names and file locations etc.

The best way to avoid misconfigurations is following best practice all the time. These best practice maybe suggested by different vendors or security experts, you should understand what the configurations are about before applying them. Implementing the configurations blindly may be unable to increase security, or even cause the application malfunctioning.

Configurations Explanation

Before you change the related configurations, you should have an idea of where the configuration file is and what different options means. Below are locations and permissions of the related files, please fully understand the content before you proceed.

/etc/apache2/apache2.conf (Apache configuration files)

This is the file responsible for configuring the Apache web server. It has a lot of settings and options including the location of root directory, logging format and session management. Usually this file is set with default configurations when the user installed the Apache package. Besides, there are multiple custom modules which users can install or enable for different functions, such as additional log formatting, advanced security controls etc. It is up to users to decide and configure their server and manage the security of it. The configuration file used here is not very complicated compared to the commercial file used in actual web application. You should spend some time to understand the options and configurations in the file if you are interested.

Allowing the server to follow symbolic links (line 170-180):
170<Directory /var/www/>
171# Options Indexes FollowSymLinks
172 AllowOverride None
173 Require all granted
174</Directory>
175
176<Directory /var/www/html/A05-misconfiguration>
177 Options Indexes FollowSymLinks
178 AllowOverride None
179 Require all granted
180</Directory>
This snippet is part of the configuration file which related to this directory. We can notice that the only difference between the two directories is the line Options Indexes FollowSymLinks being commented out or not. The first directory is the parent directory which includes other OWASP categories directories as well, while the second directory is this directory. Thus we know by enabling Options Indexes FollowSymLinks will expose the content in the whole directory. According to the documents of Apache, we know that:

  • Options means configure what features are available in this directory
  • Indexes means returning a formatted list of the directory when there is no index files exist, like index.html or index.php
  • FollowSymLinks means follow the symbolic links in this directory
Therefore this is the option which lists all the content of this directory.


testing-account.txt (Testing account information)

These are the files which store the username and the password of a testing account. It is a very common practice to have testing accounts when developing web applications, they can help to test different permissions control, authorizations and so on. However this type of resources should be strictly restricted to the scope of testing environment and must not exist in production environment. Moreover the permissions of these files are too loose such that basically anyone can read the account information of them, which is not secure at all.

testing-account.txt (permissions):
$ ls -al testing-account.txt
-rw-r--r-- 1 user user 61 Jul 16 13:10 testing-account.txt
../resources/A05/testing-account.txt (permissions):
$ ls -al ../resources/A05/testing-account.txt
-rw-r--r-- 1 user user 61 Jul 16 13:19 ../resources/A05/testing-account.txt
These are the permissions setting of the testing accounts files. The permissions of them are the same, which is -rw-r--r--. We know that the permissions of a file is set according to owner, group and others, while each part has three bits which represent read, write and execute permission of the file. Judging from the permissions we see here, it is clear that both testing accounts files have granted read permissions to everyone on the server. Hence the web server can also host the files and let everyone read the file content without any authorizations.


login.php (Emitting debug messages)

These is the file performing login checking and send the debug messages. The log in validation in this file is hardcoded for simplicity, unlike other categories which will use database for logging in. Debug messages and error messages are very useful when developing and enhancing applications, they can let the developers know the current state of the program, what is the value of each variables etc. However these messages can sometimes leak key information of the code and increase the security risks.

Defense Explanation

After knowing what the configurations are doing and how does the vulnerabilities occur, we can start to know more about how to fix them. Please try to change the code base on your understanding at least once before directly looking into the solutions. You should retry the above exploitations after you have fixed the code to find out whether you have done it correctly or not.

Vulnerability

The vulnerability exists because of a series of misconfigurations which leaks the account information. The fix of this issue is changing the configurations and following the best practices for configuring the web server. Also it is important to remove any unnecessary files from the server directory, no matter you think they will expose any information or not. Something you find useless may not be the same for others. Moreover you can explore more possible options on the Internet and enhance the security level further than the suggested solution.

Solution

/etc/apache2/apache2.conf (line 176-180):
176<Directory /var/www/html/A05-misconfiguration>
177# Options Indexes FollowSymLinks
178 AllowOverride None
179 Require all granted
180</Directory>
Comment out the line Options Indexes FollowSymLinks and make listing out the directory become impossible. This can prevent the attackers to gain extra amount of information of the server and hence less security risks. You can also implement an index page to prevent someone entering the directory page.


testing-account.txt:
$ sudo chmod 600 testing-account.txt
$ ls -al testing-account.txt
-rw------- 1 user user 61 Jul 16 13:10 testing-account.txt
../resources/A05/testing-account.txt:
$ sudo chmod 600 ../resources/A05/testing-account.txt
$ ls -al ../resources/A05/testing-account.txt
-rw------- 1 user user 61 Jul 16 13:19 ../resources/A05/testing-account.txt
Change the permissions of the files to 600 can prevent anyone other than the owner to read the testing accounts files. Since each permissions can be represented by a binary number, which read is 100 (decimal: 4), write is 010 (decimal: 2) and execute is 001 (decimal: 1), we can grant the original permissions to the owner, which is 6 (4+2) and we grant no permissions (0) to the others users. You can also delete the testing account files, which is the best option, if you do not need them at all.


login.php (line 9-18):

10$debug = "Incorrect credentials";
11

13 $message = "Login succeed";

16 $message = "Login failed";

Changing the debug message to hide the location of the testing accounts information is only a short-term measure here. There should be a proper mechanism to create and request testing accounts information for different purposes in the long-term. Also you should always keep the debug messages precise and avoid providing personal information, such as the email of the IT department in this case.
